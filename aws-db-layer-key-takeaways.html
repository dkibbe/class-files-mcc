Adding a Database Layer on AWS

Key Takeaways

AWS Academy Cloud Architecting — Module 06

What This Module Enables You To Do

  • Compare relational vs. non‑relational databases and AWS services.
  • Identify when to use Amazon RDS and when to use Amazon DynamoDB.
  • Recognize AWS purpose‑built databases and common workloads.
  • Understand migration options (AWS DMS, AWS SCT).
  • Apply AWS Well‑Architected principles to the database layer.

Database Layer Considerations

  • Scalability: anticipate throughput; choose vertical, horizontal, or hybrid scale.
  • Storage: size for GB/TB/PB and growth.
  • Data characteristics: model, access patterns, and latency requirements drive selection.
  • Durability & compliance: availability, recovery, and regulatory needs.
  • Relational vs. non‑relational: strict schema/vertical scale vs. flexible schema/horizontal scale.
  • Managed services: offload undifferentiated heavy lifting; focus on application optimization.

Amazon RDS — Essentials

  • Managed relational service supporting multiple engines (incl. Aurora).
  • Benefits: lower admin overhead, elastic scaling, high availability, strong security.
  • Amazon Aurora: MySQL/PostgreSQL‑compatible, cloud‑built performance and availability; Serverless for on‑demand capacity.
  • Sizing: pick instance families for CPU vs. memory needs.
  • Security: VPC isolation, IAM, security groups, TLS, KMS, engine‑level controls.

RDS Proxy & Data Protection

  • RDS Proxy: connection pooling, faster failovers, IAM auth, Secrets Manager creds.
  • Availability patterns:
    • Multi‑AZ: synchronous replication & automatic failover.
    • Read replicas: async replication for read scaling.
  • Backups & Encryption: automated backups/snapshots; encrypt at rest (KMS) & in transit (TLS). Snapshot‑copy to encrypt legacy DBs.

Amazon DynamoDB — Essentials

  • Fully managed, serverless NoSQL (key‑value & document) with single‑digit ms latency.
  • Model: tables, items, attributes; simple vs. composite keys.
  • Indexes: GSI (eventual) & LSI (eventual/strong) for flexible queries.
  • Global tables: multi‑Region, multi‑active replication for global apps.
  • Resilience & security: encryption by default; point‑in‑time recovery; fine‑grained IAM; CloudTrail/Config.

Purpose‑Built Databases

  • Amazon Redshift: analytics (OLAP), columnar, serverless option.
  • Amazon DocumentDB: MongoDB‑compatible JSON documents.
  • Amazon Keyspaces: Cassandra‑compatible wide‑column.
  • Amazon MemoryDB for Redis: durable in‑memory with ultra‑low latency.
  • Amazon Neptune: graph for highly connected data.
  • Amazon Timestream: time‑series workloads.
  • Amazon QLDB: immutable, cryptographically verifiable ledger.

Migrating Data into AWS Databases

  • AWS Database Migration Service (DMS): managed migration/replication; homogeneous & heterogeneous.
  • Schema conversion: AWS SCT & DMS Schema Conversion for code/schema translation.
  • Use cases: move to RDS/Aurora; replicate to data lakes; scheduled or CDC pipelines.

Well‑Architected for the DB Layer

  • Performance efficiency: align engine/config with data characteristics and access patterns.
  • Security: protect data at rest/in transit; enforce strong key management.
  • Cost optimization: select resource types/sizes/count based on measured demand and usage.

Hands‑On Focus

  • RDS lab: provision a DB, connect an app, practice secure/HA settings.
  • Migration lab: import data from EC2 to RDS and retarget application connectivity.

Closing

  • Choose the right database for workload, data model, and growth.
  • Prefer managed services to accelerate delivery and improve resilience.
  • Design for availability, security, performance, and cost from day one.
